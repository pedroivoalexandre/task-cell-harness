# Local Artifacts Policy

## Purpose

This document defines which files are local artifacts and how the Task Cell Harness should treat them.
The goal is to reduce repository noise, improve portability, and keep GitHub as the source of truth.

## What Counts As A Local Artifact

Local artifacts are files or directories created by running tests, validations, or local harness commands.
They are useful during execution, but they are not part of the authoritative project state.

Typical examples:

- Python bytecode caches;
- local logs;
- runtime locks;
- runtime state files;
- temporary runtime output;
- validation output generated on the developer machine.

## What Should Be Tracked

The repository should track:

- source code;
- tests;
- documentation;
- versioned configuration;
- stable task contracts;
- intentionally curated examples and fixtures.

If a file is part of the official project definition and should exist identically across machines, it belongs in Git.

## What Should Be Ignored

The repository should ignore:

- `__pycache__/`;
- `*.pyc`;
- local logs;
- `artifacts/`;
- `runtime/locks/`;
- `runtime/state/`;
- `runtime/temp/`;
- common temporary files;
- local OS metadata files such as `.DS_Store` and `Thumbs.db`.

These paths are transient and should not pollute diffs or commits.

## Logs

Logs produced by local execution are operational output, not project source.
They should normally be ignored.

If a log file is required as a stable fixture or reference sample, it should be documented explicitly and isolated from normal execution output.

### `logs/harness.log`

`logs/harness.log` should be treated as a local operational log unless a future decision explicitly reclassifies it as a curated fixture.
Until then, it should not be used as a source of project truth.

## Python Caches

Python cache directories and `.pyc` files are generated automatically by the interpreter.
They should be ignored, because they are machine-specific, version-specific, and reproducible at any time.

## Runtime Directories

The following paths are local and ephemeral:

- `runtime/locks/`
- `runtime/state/`
- `runtime/temp/`

These directories are for runtime coordination and should not be treated as versioned project state.

## Artifacts

`artifacts/` is the local output area for execution and validation results.
It should remain outside version control unless a specific artifact is intentionally promoted to a tracked fixture or document.

## Reports

`reports/` is mixed-use:

- canonical or curated reports may be tracked;
- generated validation output should be treated carefully.

The guiding rule is whether the file is an intentional, stable project reference or a disposable execution result.
If it is disposable, it should not be committed.

## Task Logs

`logs/tasks/*.jsonl` may be used for operational traceability.
These files should be tracked only if they are intended as canonical fixtures or durable project records.
If they are generated by routine execution, they should be ignored or moved into a dedicated non-versioned output area.

## Fixtures And Canonical Examples

Canonical fixtures should live in dedicated, clearly named locations.
They should not be mixed with transient runtime output.

Preferred locations for canonical material:

- dedicated test fixture directories;
- documentation examples;
- versioned configuration directories;
- explicit sample data paths.

## Portability Impact

This policy improves portability across:

- Lenovo Yoga as the primary development worker;
- Galaxy Tab S10+ as the future Control Plane;
- Acer as a future worker;
- XPS13 as a future worker;
- Linux, Windows, and Android/Termux environments.

Ignoring local artifacts reduces environment-specific diffs and makes the repository easier to clone, run, and validate on each machine.

## GitHub As The Source Of Truth

GitHub remains the official source of project state.
Local artifacts are disposable unless they are explicitly promoted into tracked project assets.

## Technical Recommendation For A Later Step

In a later step, with explicit authorization, the following candidates should be reviewed for removal from tracking if they are confirmed to be local artifacts:

- `__pycache__/`
- `*.pyc`
- `logs/harness.log`
- any other locally generated log files that are not canonical fixtures;
- any runtime artifact that is currently tracked but should be local-only.

That later step should use explicit tracking changes and must not be conflated with this policy document.
